Executive Summary
The short answer: you cannot generate arbitrary-text-to-image in a single fully-bundled HTML file with zero downloads, if by "text" you mean free-form natural language prompts like Stable Diffusion accepts. Every real text-conditional generative model requires a text encoder (CLIP/T5) that alone weighs 250–500 MB minimum. However, there are four viable paths depending on your definition of "offline" and the scope of prompts you need to support. One of them — pre-baking a small domain-specific model as Float32 weights inside the HTML — is genuinely feasible at ~1–5 MB for closed-set prompts (up to ~50 categories). And yes, you can absolutely pre-train your own model, on consumer hardware, for a narrow domain.
01

The Core Constraint Explained

Real text-to-image generation requires two fundamentally separate things: a text understanding component (to turn "a red balloon over a snowy mountain" into a numeric vector), and an image generation component (to turn that vector into pixels). Both require pre-trained knowledge baked into weights.

The text encoder is the binding constraint. Even the smallest useful CLIP text encoder (ViT-L/14) weighs ~470 MB. The smallest T5 encoder used by modern diffusion models is ~900 MB. You cannot compress these below ~80 MB without losing the ability to understand general language — because they must have learned associations between millions of concept words and visual meanings during training on internet-scale data.

Hard Floor
For truly arbitrary free-form text prompts (like "a photorealistic red apple on a marble countertop in soft morning light"), the minimum model weight bundle that produces recognizable output is approximately 250 MB (achieved in 2025 with 6-bit/8-bit ONNX quantization of SD ControlNet). There is no path below this for general text.

However, if you constrain the problem — e.g. "generate an image from a menu of 20 categories with adjectives" — the required model drops to kilobytes, because you don't need to understand arbitrary language anymore. You need to understand a vocabulary of ~50 tokens, which a tiny embedding table can encode.

Approach A — Pre-Bake Weights Directly into the File

This is the most interesting path and what µ²Diffusion v3 implements. The idea: train a tiny domain-specific model offline in Python, export its weights as a Float32Array literal hardcoded into the JavaScript, deploy as a single HTML file with zero dependencies.

What's Feasible

Key insight
When the text vocabulary is closed (e.g., a structured prompt UI with dropdowns: "a [ADJ] [SUBJECT] [STYLE]"), you don't need a general language model. A learned embedding table for 200 tokens fits in under 50 KB. The entire generative model — encoder + UNet + decoder — can fit in under 5 MB for 64×64 images.

The Bundling Technique

// Python: export trained PyTorch weights
import torch, json, base64
state = model.state_dict()
flat = torch.cat([p.flatten() for p in state.values()])
b64 = base64.b64encode(flat.numpy().tobytes()).decode()
with open('weights.js', 'w') as f:
    f.write(f'const WEIGHTS = "{b64}";')

// JavaScript: load at runtime
const bytes = Uint8Array.from(atob(WEIGHTS), c=>c.charCodeAt(0));
const weights = new Float32Array(bytes.buffer);

A 1 MB model becomes a 1.33 MB base64 string. A 5 MB model becomes 6.67 MB. Both are perfectly reasonable for a single HTML file — browsers routinely handle JS files of this size.

04

Approach B — Train in the Browser (What µ²Diffusion v3 Does)

Instead of baking pre-trained weights, you can train a tiny model inside the browser on first launch, using procedurally generated training data. This is what the current µ²Diffusion HTML file does.

Advantages: The file is tiny (~60 KB). No weights to bundle. The user experiences "AI training live in your browser." Training data is infinite and free (procedural generation).

Disadvantages: Training takes 30–90 seconds on first launch. Pure JS training is ~50–200× slower than PyTorch on GPU, so we're limited to small models and simple data. The model only knows what the procedural generator teaches it — it cannot learn photorealistic texture or complex real-world scenes this way.

Performance reality check
A 100K-parameter UNet in pure JavaScript (no WebGPU, no WASM ML library) does about 10,000 forward passes per second for 32×32 images. This is enough to run 3,000 training steps in ~60s. A proper 1M-parameter model would take ~10 minutes. WebGPU (available in Chrome/Edge/Firefox) would bring this to near-PyTorch speed, potentially enabling 64×64 real-data training in the browser in future.

WebGPU Changes the Equation

As of 2024–2025, WebGPU is now available by default in Chrome 113+, Edge 113+, and Firefox 141+, covering approximately 85% of desktop browsers. With WebGPU compute shaders, you can achieve genuine GPU-accelerated training inside the browser. Microsoft's ONNX Runtime Web + WebGPU can run Stable Diffusion Turbo in under 1 second on an RTX 4090 inside a browser tab.

This opens a future path: bundle a tiny WASM training engine, and train a compact domain model in the browser using WebGPU in 30 seconds instead of 90 seconds — and do it at 64×64 instead of 32×32.

Approach C — Procedural / Non-Neural Image Synthesis

Before deep learning, text-to-image systems worked by parsing text into scene descriptions and rendering them programmatically. This is what µ²Diffusion v1 did (the canvas procedural renderer). This approach has a long research history going back to 2000s "clip art assembly" systems.

SVG Path Grammar
Text → parametric SVG shapes. Compact, resolution-independent, offline. Supports compositing (sky, ground, foreground objects) from text tokens.

Pre-deep-learning history
The first text-to-image research (alignDRAW, 2015) produced 32×32 images that were "low in diversity" but could handle novel prompts like "a stop sign flying in blue skies." This was before DALL-E, Stable Diffusion, or even GAN-based T2I. Non-neural synthesis of comparable quality to early 2015 systems is entirely achievable in pure JS today.
06

Approach D — Tiny Domain-Specific GAN (Pre-Train, Bundle, Deploy)

The most practical path for a domain-specific product: train a DCGAN or conditional DDPM on a specific narrow dataset (flowers, faces, icons, landscapes, anime characters) in Python, then export the generator only as hardcoded weights in JavaScript. The discriminator is discarded after training.

Why this works well

A DCGAN generator for 64×64 images has approximately 3.5 million parameters. In float32, that's 14 MB. Quantized to int8, it's 3.5 MB. Compressed with gzip in the HTML, it's ~2 MB. This is completely practical to bundle in a web application.

The training pipeline uses a DCGAN or lightweight UNet where the text/class conditioning is done via a simple lookup table (50 class labels → 50 learned 128-d vectors). No CLIP needed. No T5 needed. Just: "which category?" → embedding → condition the generator.

Proven approach — real GAN-based T2I in 2016
Reed et al. 2016 ("Generative Adversarial Text to Image Synthesis") trained a GAN on CUB bird descriptions and Oxford flower descriptions with a character-level text encoder. The model could generate "visually plausible 64×64 images" from text like "a flower with yellow petals and a round purple center." The entire model — generator + text encoder — was under 50 MB. This is still the baseline for what's achievable in a bundleable approach.

This is the current absolute minimum for general-purpose free-form text-to-image in a browser, still requiring a first-run download.

Can We Pre-Train Our Own Model?

Yes, absolutely. The question is: what kind of model, on what data, and how long does it take?

Option 1: Train a Domain DCGAN from Scratch

A DCGAN at 64×64 resolution with class conditioning on a dataset of ~5,000–50,000 images is entirely trainable on a consumer GPU in hours. The PyTorch DCGAN tutorial uses the CelebA dataset (~200K faces at 64×64) and trains to recognizable output in ~5 hours on a single RTX GPU. For a narrower domain (icons, simple illustrations, faces) with 5,000 images, expect 2–8 hours.

Option 2: Fine-tune BK-SDM-Tiny with DreamBooth

BK-SDM-Tiny (ECCV 2024) is a block-pruned, knowledge-distilled SD model with a 330 MB UNet. It was trained on only 0.22 million LAION pairs — 13 A100 GPU-days. You can fine-tune it with DreamBooth on a custom domain (specific art style, product, character) in ~1 hour on an RTX 3090 with as few as 20–30 images. The result responds to text prompts about your domain with high quality. Total deployment package is still ~2 GB though — so not bundleable.

Option 3: Train a Tiny Conditional DDPM from Scratch (What µ²Diffusion Does)

This is the in-browser approach. A ~400K parameter UNet at 32×32 with 10 class conditions, Adam optimizer, 3,000 steps — trains in the browser in 60 seconds. In Python with PyTorch on GPU, the same model trains in under 30 seconds and produces weights you can export as Float32 literals. Quality matches what DALL-E produced circa 2015 for domain-specific prompts.

Option 4: Knowledge Distillation from SD

If you have a GPU and time, you can distill Stable Diffusion into a smaller student model following the BK-SDM approach. Segmind released an open-source distillation toolkit that produces SD-Small and SD-Tiny variants. Training requires 13+ A100-days, but you can approximate it with consumer GPUs over several weeks. The result is a ~500 MB model that accepts free-form text — still not bundleable, but deployable as a one-time-download cache.

Transfer learning shortcut
Rather than training from scratch, you can adapt a pretrained GAN (e.g., ImageNet-pretrained SNGAN or BigGAN) to a custom small dataset by updating only scale+shift parameters — requiring as few as 100 images and a few minutes of training. This is described in the "Image Generation From Small Datasets via Batch Statistics Adaptation" paper (ICCV 2019). The downside: you still ship the full pretrained generator weights (~200–500 MB for BigGAN), just update very few additional parameters.

The data problem
Training a general text-to-image model requires billions of (image, caption) pairs covering all visual concepts. This is why nobody has trained a general model from scratch since 2022 — the data collection and compute cost is prohibitive. The only practical path to "general free-form text" is fine-tuning an existing open model like SD 1.5, SDXL, or BK-SDM-Tiny.

Recommended Strategy for This Project

Based on the research, here are the four realistic paths, ordered from "most bundleable" to "most powerful":

Path 1: Domain DCGAN — pre-trained Python → bundled JS
Train a 64×64 conditional DCGAN in PyTorch on a real dataset (your domain). Export generator weights as Float32Array. Bundle directly into the HTML. Result: single HTML file, instant generation, no downloads.

Requires: RTX GPU, real images for your domain, 4–24 hours of training. Produces: recognizable domain-specific images from ~20–50 text labels.
FULLY BUNDLED REAL IMAGES FIXED VOCABULARY
Path 2: Enhance µ²Diffusion (current) — richer procedural data
Keep the in-browser training approach but improve the procedural data generator: more realistic shapes, texture simulation, color theory. The 32×32 results will be more polished. Pure JS, no new infrastructure needed.
NO PYTHON NEEDED IN-BROWSER TRAINING SYNTHETIC ONLY
Path 3: Train in Python, serve via one-time cache
Like Path 1 but use the Transformers.js (ONNX) pipeline with a fine-tuned BK-SDM-Tiny on your domain. First load is ~500 MB cached to IndexedDB. After that: fully offline, responds to free-form text. This is the µ²Diffusion v2 approach, but with a custom-trained model.
ONE-TIME 500 MB DL FREE-FORM TEXT HIGH QUALITY
Path 4: 250 MB bundled SD ControlNet (2025 approach)
Use Lee Butterman's 6/8-bit quantized SD ControlNet in ONNX format. Deliver as a ~250 MB download, cached forever after first visit. Best quality-to-size ratio for general free-form text currently achievable in a browser.

Bottom Line

For "zero bytes ever downloaded" + "real images" + "natural language text": not currently possible — the physics of language understanding require weights.

For "zero bytes ever downloaded" + "real images" + "structured/keyword prompts": YES — Path 1 achieves this. Train in Python, bundle into HTML. 64×64 recognizable images, ~50 text categories, single HTML file under 15 MB.

For "first-run download, then fully offline" + "free-form text": YES — our current v2 (Transformers.js) or Path 4 (250 MB quantized). After first cache, operates identically to a bundled file.
